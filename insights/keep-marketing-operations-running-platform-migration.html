Marketing operations

How to keep marketing operations running during a platform migration

The safest migration plan treats day-to-day campaign work as a first-class delivery stream, not as noise around the project.

By Sviatoslav FoshchiiPublished June 28, 20266 min read

Marketing teams rarely get a quiet quarter for a platform migration. Campaigns still need to launch, stakeholders still need answers and production issues still appear. That is why operational continuity has to be designed into the migration from the start.

Business-as-usual is not a distraction from a migration. It is the environment the migration has to survive.

Separate migration work from operations work

I like to make two lanes visible: migration delivery and live operations. The migration lane contains discovery, build, QA, data mapping, documentation and handover. The operations lane contains active campaigns, urgent fixes, stakeholder support, approvals and production monitoring.

When both lanes are visible, the team can make better tradeoffs. When only migration tasks are visible, operations work becomes invisible pressure.

Define what cannot break

Not every campaign, automation or report carries the same risk. Before the migration gets busy, define the non-negotiables: critical customer journeys, revenue-sensitive sends, compliance-related communications, peak-season campaigns and escalation paths.

That list becomes a practical filter. It helps the team decide where to add extra QA, where to slow down and where a simpler temporary process is acceptable.

Keep a shared decision log

Migrations create hundreds of small decisions. If those decisions live only in meetings, the team will repeat conversations and rediscover old assumptions.

A useful decision log is simple: date, topic, decision, owner, reason and follow-up. It does not need to be perfect. It needs to be searchable and trusted.

Make QA boring and repeatable

Good QA is not hero work. It is a repeatable checklist that reduces uncertainty: data source confirmed, audience logic reviewed, personalization tested, links checked, rendering reviewed, suppression logic confirmed, tracking validated and rollback path understood.

When QA is consistent, stakeholders learn what "ready" means. That makes approvals faster and reduces subjective last-minute changes.

Plan the handover before the end

Handover should not wait until the last week. The receiving team needs time to absorb the new way of working, challenge unclear documentation and rehearse support scenarios.

For me, a good handover includes ownership, known limitations, support routines, naming conventions, QA checklists and examples of common changes.

My practical takeaway

The best way to protect marketing operations during a migration is to treat operational work as part of the plan. That sounds obvious, but it is where many migration plans become unrealistic.
